# 高俊杰个人收获与反思

## 1. 我在项目中的主要工作

在这个项目中，我主要承担组长、工程架构、代码统筹和最终整合的工作。虽然项目是小组合作完成，但从实际推进过程看，我负责了绝大部分最终提交材料的组织和完善，包括 GitHub 仓库结构、README、复现入口、M1-M3 报告整合、图表逻辑检查、3 图 1 表补齐、AI 使用说明、参数说明、reflection 目录和答辩材料归档。

我最主要的工作不是单独做某一张图，而是把项目从“有数据、有脚本、有图片”的状态整理成一个能被老师快速巡检、能说明研究问题、能复现核心结果、能支撑 mini review 的完整 GitHub 项目。

具体来说，我完成或主要推动了以下工作：

1. 设计并整理项目目录结构，让 `config/`、`data/`、`src/`、`outputs/`、`reports/`、`paper/`、`presentation/` 和 `reflection/` 各自承担清晰功能。
2. 整合 `run_pipeline.py`，让项目能够通过统一入口运行数据质量检查、趋势分析、指标计算、网络分析和敏感性分析。
3. 多次重写 README，使其包含研究主题、成员分工、数据来源、筛选流程、RQ、3 图 1 表、Top 10 高被引文献表、图表索引、复现命令、课程交付物和局限性说明。
4. 检查 M2 和 mini review 中图表与文字的逻辑关系，避免图像只是被插入，而没有服务于结论。
5. 补充 `params.md`、`docs/ai_usage.md`、`data/field_dictionary.md`、`data/screening_records.csv` 和本 reflection 目录，使仓库更符合 Lesson 27 & 28 的巡检要求。
6. 整理答辩 PPT 的归档路径，确保 README 能直接指向答辩材料。

## 2. 我对文献计量学的理解变化

项目开始时，我对文献计量学的理解比较简单，觉得它主要就是统计发文量、画关键词网络图、做作者合作图和引用图。真正做完整个项目以后，我发现文献计量学最重要的不是“图多不多”，而是每张图是否回答了一个明确的问题。

比如年度发文趋势图并不是为了说明“这个领域很火”这么简单。更严谨的说法是，它可以说明 2024-2025 年文献层面的研究活动明显增加，提示该领域进入快速扩张阶段；但它不能证明技术已经成熟，也不能证明产业应用已经落地。这个边界意识是我在项目后期反复修改 README 和 mini review 时才真正理解的。

关键词共现图也是类似。之前我可能会写“图中出现了 artificial intelligence、large language models、agentic ai 等关键词”，但这样的描述只是把图上内容读了一遍。后来我学会了把关键词组织成技术底座、协作架构、能力增强、治理应用四个层次，这样图表才真正服务于综述结构。

## 3. 我对 GitHub 可复现项目的认识

这次项目让我意识到，GitHub 不是网盘，不能只是把文件堆进去。一个合格的课程项目仓库要让别人一打开就知道：

- 研究主题是什么。
- 数据从哪里来。
- 检索式是什么。
- 原始数据如何筛选成最终样本。
- 使用了哪些参数和阈值。
- 如何运行代码。
- 输出结果在哪里。
- 图表如何支撑论文结论。
- 小组如何分工。
- 个人贡献和反思在哪里。

我在 README 中加入“GitHub 巡检材料导航”，就是为了让老师在两分钟闪检中可以直接找到材料。这个过程让我认识到，项目可复现不仅是代码能跑，还包括文档清楚、路径稳定、参数透明和证据链完整。

## 4. 我对团队协作的反思

作为组长，我最大的体会是，项目制学习最难的不是某个具体任务，而是最后把所有人的工作整合成一个统一的成果。前期大家分别做检索、清洗、图表、综述时，每个部分都有内容，但这些内容不一定天然连贯。

后期我花了很多时间做整合，包括：

- 统一文件命名和目录结构。
- 检查 README 与报告是否一致。
- 把 RQ 和图表重新对应起来。
- 把高被引论文表补完整。
- 给图表解释增加边界说明。
- 补齐 reflection、AI 使用说明和参数说明。

我也认识到自己前期做得还可以更好。如果项目一开始就建立更严格的任务表，比如 `method_result_map.csv`、`figure_question_map.csv` 和 `claim_evidence_table.csv`，后期返工会少很多。以后再做小组项目，我会更早建立文件规范和证据链表，而不是等到最后才集中整理。

## 5. 我对 AI 辅助学习的反思

这个项目中我使用了 AI 辅助进行 README 结构整理、文字润色、图表解释逻辑检查、代码问题排查和答辩材料组织。但我也认识到，AI 不能替代人工判断，尤其不能替代真实数据和文献证据。

例如，发文量、被引次数、h 指数、References 覆盖率、Top 10 高被引论文这些内容，都必须来自项目数据和脚本输出，不能由 AI 编造。AI 可以帮我发现“这里缺少边界说明”“这个图和 RQ 对应不够清楚”“README 没有 reflection 入口”，但最终判断必须由我回到本地文件核验。

我觉得比较合理的 AI 使用方式是：把 AI 当作结构检查、语言组织和代码辅助工具，而不是事实来源。这个认识也体现在我补充的 `docs/ai_usage.md` 中。

## 6. 我的主要收获

通过这个项目，我最大的收获有四点。

第一，我学会了如何把一个宽泛主题收束成可回答的 RQ。比如“LLM 多智能体协作研究趋势”很大，但可以拆成发展态势、合作格局、研究热点、知识基础和未来挑战。

第二，我学会了图表和结论之间必须建立明确关系。不是有图就行，而是图要回答问题，问题要服务综述，综述要有证据链。

第三，我学会了 GitHub 项目需要工程化。一个好的 README、清晰目录、稳定运行入口和参数说明，能大幅提高项目可信度。

第四，我对学术表达中的谨慎性有了更深认识。文献计量结果主要反映文献层面的关注度、结构和趋势，不能随便推出技术成熟、应用落地或未来必然主流。

## 7. 未来还需要学习的内容

后续我希望继续学习以下内容：

1. 更系统地学习 CiteSpace 和 VOSviewer，补充 burst detection、centrality、cluster label 等更专业的科学知识图谱指标。
2. 学习更规范的 systematic review 写法，包括 PRISMA 流程、纳入排除标准、证据表和参考文献管理。
3. 提高 Python 数据分析能力，让脚本结构更加模块化，减少后期手工整理。
4. 学习如何写更专业的图注，尤其是数据来源、时间范围、分析单位、工具参数、节点大小、颜色和连线含义。
5. 学习如何在小组项目早期建立任务追踪和版本规范，避免最后集中补材料。

## 8. 总结

这次项目让我真正理解了文献计量型 Review 的核心：不是简单堆文献，也不是单纯画图，而是用透明的数据和方法，把图表解释成有边界、有证据、有逻辑的学术判断。

作为组长，我在这个项目中承担了大部分最终整合和提交工作。虽然过程比较赶，也有很多返工，但我确实学到了如何把课程要求、代码、图表、报告和答辩材料组织成一个完整项目。这个经验对我以后做科研训练、课程项目和 GitHub 作品集都有很大帮助。
