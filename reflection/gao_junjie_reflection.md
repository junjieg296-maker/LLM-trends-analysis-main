### 2.1 高俊杰的主要贡献

高俊杰在项目中承担的任务不是单一模块，而是从项目启动到最终提交的整体统筹，具体包括以下几个方面。

#### 2.1.1 项目结构与 GitHub 归档

我负责把原本分散的脚本、数据、图表和报告整理成适合 GitHub 提交的项目结构。项目最终形成了比较清晰的目录分层：

- `config/`：保存检索式和参数化配置。
- `data/`：保存原始数据、清洗数据、字段字典和筛选记录。
- `src/`：保存数据清洗、指标计算、网络分析和图表生成脚本。
- `outputs/`：保存指标汇总、关键词统计、milestone 候选论文和图表输出。
- `reports/`：保存 M1、M2、M3 阶段报告。
- `paper/`：保存 mini review 终稿。
- `presentation/`：保存答辩材料说明。
- `reflection/`：保存个人贡献与反思材料。

这一部分的工作看起来像“整理文件”，但实际上决定了项目能不能被老师快速巡检。Lesson 27 & 28 明确要求 GitHub 仓库要能回答三个问题：材料在哪里、能否复现、下一步修什么。因此，我在 README 中补充了研究主题、成员分工、数据来源、运行命令、输出目录说明、3 图 1 表、RQ 与图表对应关系、课程交付物和局限性说明。

#### 2.1.2 自动化复现入口

我参与并统筹了 `run_pipeline.py` 的整合，使项目不只是若干独立脚本，而是可以通过一个入口串联主要分析步骤。当前流水线包括：

1. M1 数据质量检查。
2. 年度发文趋势图生成。
3. 标题词云生成。
4. M2 计量指标计算。
5. milestone 候选论文识别。
6. 关键词共现网络构建。
7. 作者合作网络构建。
8. 机构 / 国家分布图构建。
9. 文献共被引网络构建。
10. 引用网络构建。
11. 参数敏感性分析。

这一部分对应课程中的“可复现 GitHub 仓库”要求。相比只提交图片，自动化入口能够说明图表不是手工拼出来的，而是可以从数据、脚本和参数重新生成。

#### 2.1.3 README 与证据链完善

README 是老师打开 GitHub 后最先看到的材料，因此我重点完善了 README。主要补充包括：

- 将项目状态更新为 M3 完成版。
- 写清楚研究主题、数据来源、检索策略和筛选流程。
- 补充 RQ 设计，并按 Lesson 21 & 22 的要求将 RQ 与图表、指标、解释和边界绑定。
- 明确写出课程要求中的 3 图 1 表。
- 将 Table 1 扩展为 Top 10 高被引文献表。
- 补充 M1-M3 阶段成果与图表索引。
- 补充主要结论、局限性说明和课程交付物。

在这个过程中，我最大的工作量不是简单增加文字，而是反复检查“图是不是为了插而插”“每个结论有没有证据”“每张图能说明什么、不能说明什么”。这也对应 Lesson 22 中强调的证据链模型：Claim、Bibliometric Evidence、Literature Evidence、Interpretation 和 Boundary。

#### 2.1.4 图表逻辑与综述结构修订

前期项目中的一些图表虽然已经生成，但存在“图像和文字逻辑关系不够紧”的问题。后续我重点做了两类修订：

1. **M2 报告修订**：将引用网络、共被引网络和 milestone 候选论文之间的关系重新理顺。引用或高被引图用于回答“哪些论文更有影响力”，共被引网络用于回答“这些论文背后有哪些共同知识基础”。
2. **综述修订**：把关键词主题、milestone 候选论文和共被引网络转化为四条技术叙事线，而不是简单堆放图片。

这部分工作帮助项目从“有图表”变成“图表服务于论证”。我认为这是文献计量型 Review 最关键的一步，因为图谱本身不是综述，只有经过解释、归纳和边界说明，图谱才真正成为论文证据。

#### 2.1.5 答辩材料准备

我还参与整理了答辩 PPT 的结构和内容。答辩材料围绕课程要求、研究问题、数据口径、3 图 1 表、核心发现、局限、GitHub 路径和分工完成度展开。虽然 PPT 不是本 reflection 文档的重点，但它促使我重新检查了 README、报告和图表之间的对应关系，避免答辩时出现“PPT 说了，但仓库找不到”的问题。

## 3. 高俊杰个人收获与反思

### 3.1 对文献计量学的理解变化

在项目开始之前，我对文献计量学的理解比较浅，更多把它看成“统计发文量、画关键词图、做一些网络图”。做完这个项目后，我意识到文献计量学真正重要的不是图本身，而是图表背后的研究问题和证据链。

例如，年度发文趋势图不能简单写成“发文量增加，所以该领域很重要”。更严谨的写法应该是：年度发文量显示该领域在 2024-2025 年进入快速扩张阶段，这说明文献层面的研究关注度显著上升；但这不能直接证明技术已经成熟，也不能证明应用已经充分落地。这个“能说明什么”和“不能说明什么”的区分，是我在课程后半段才逐渐建立起来的。

同样，关键词共现图也不能只写“图中有很多节点”。它要回答的是研究热点和主题结构。通过这个项目，我学会了把关键词共现结果解释为技术底座、协作架构、能力增强和治理应用四个层次，这比单纯罗列关键词更接近 Review 写作的要求。

### 3.2 对 GitHub 可复现项目的认识

这个项目让我明显感受到，GitHub 项目不是把文件上传上去就结束了。一个可复现项目至少要让别人知道：

1. 数据从哪里来。
2. 检索式是什么。
3. 为什么保留这些文献、排除哪些记录。
4. 用了哪些参数。
5. 如何运行代码。
6. 输出文件在哪里。
7. 图表如何支撑论文结论。

在整理 README 的过程中，我反复对照 Lesson 27 & 28 的巡检清单，发现一个项目如果没有清晰目录和入口，即使实际做了很多工作，也很难在两分钟闪检中讲清楚。因此，我开始把 README 当成项目的“说明书”和“证据地图”，而不是简单的项目简介。

这也是我个人贡献最多的一部分：我不断把分散材料重新组织成可检查的路径。比如老师要看数据来源，可以去 `config/query.yaml` 和 `data/README.md`；要看筛选与质量，可以去 `reports/m1_data_quality_report.md`；要看 3 图 1 表，可以直接在 README 的对应表里找到；要看高被引文献表，可以去 `outputs/milestone_paper_candidates.md` 或 README 中的 Table 1。

### 3.3 对团队协作的反思

作为组长，我最大的体会是：项目制学习中，最难的不是某一个脚本或某一张图，而是让所有材料最后能够形成同一个方向。前期我们有数据、有图、有报告，但这些东西一开始并不是天然连贯的。有时一张图已经生成了，但它到底回答哪个 RQ、支持哪一段文字、有没有图注和边界说明，并没有马上明确。

后期我做得比较多的工作，就是把这些内容重新串起来。这让我意识到组长的工作不只是分配任务，还包括：

- 统一项目叙事。
- 检查文件命名和目录结构。
- 判断哪些图可以支撑结论，哪些图只能作为辅助。
- 把组员产出的局部成果整理成完整项目。
- 在提交前检查课程要求是否逐项满足。

我也认识到自己前期可以做得更好。比如，项目开始时如果就建立 `method_result_map.csv` 或类似的 RQ-图表对应表，后续写综述和 README 会更顺利。前期如果更早规定图注格式、参数记录和表格模板，也能减少后期返工。这个经验对以后做科研项目很有帮助：项目越早建立证据链和文件规范，后期越不容易散。

### 3.4 对 AI 辅助学习的反思

这个项目中使用了 AI 辅助进行文字润色、结构检查、README 组织、图表解释逻辑检查和代码问题排查。但我也更清楚地意识到，AI 不能代替人工判断。

AI 可以帮助我发现“这段解释不够连贯”“这个图缺少说明”“README 缺少反思文件”等问题，也可以帮助我把已有材料组织成更清晰的文档。但文献数据、指标数值、图表结论和边界判断必须回到项目本身，不能凭 AI 生成。尤其是高被引论文表、发文量、h 指数、References 覆盖率这些内容，必须来自本地数据或已有报告。

这让我对“AI 使用说明”的必要性有了更具体的理解。使用 AI 并不是问题，问题在于是否说明了 AI 的用途、是否保留人工核验、是否避免编造引用和数据结论。后续我会更注意把 AI 当成校对和组织工具，而不是事实来源。

### 3.5 对综述写作的收获

通过这个项目，我对文献计量型综述的结构有了更清楚的认识。一个合格的 Review 不是“背景 + 很多论文 + 几张图”，而是应该有明确的 Introduction、Data and Methods、Results、Discussion 和 Conclusion。

其中我收获最大的是 Results 和 Discussion 的区别。Results 应该围绕图表证据说明事实和模式，比如发文量增长、关键词主题、合作网络分散、共被引主题簇。Discussion 则进一步解释这些模式意味着什么，以及未来可能有哪些挑战。但 Discussion 也不能写成没有证据的预测，而要谨慎使用“可能”“显示”“提示”等表述。

在重写 README 和 mini review 的过程中，我学会了给每个结论加上边界。例如：

- 发文增长不能等于技术成熟。
- 高频关键词不能等于未来必然主流。
- 被引高不能等于论文一定正确。
- 合作网络中心位置不能等于研究质量最高。

这种边界意识是我这门课中最重要的收获之一。

### 3.6 项目不足与后续改进计划

虽然项目已经形成了比较完整的 GitHub 成果包，但我认为仍有一些不足：

1. **部分元数据仍受 Lens.org 字段质量限制。** 国家、机构、作者字段可能存在缺失或格式不统一，后续可以增加更严格的字段标准化。
2. **共被引分析仍属于课程项目级别。** 后续如果时间充足，可以使用 CiteSpace 或 VOSviewer 补充 burst detection、centrality、cluster label 等更专业指标。
3. **代表文献阅读还可以更深入。** 当前 Top 10 高被引论文已经列出，但每篇文献的理论贡献、方法贡献和应用场景还可以做成更详细的 evidence table。
4. **参数记录可以进一步结构化。** 虽然已经有 `params.md` 和敏感性分析报告，但后续可以把每个图表的参数、阈值、输入文件和输出文件整理成机器可读表格。
5. **答辩材料可以继续压缩。** 目前项目材料较完整，但正式答辩时间有限，需要在 PPT 中更精简地讲清楚 RQ、数据、3 图 1 表、结论和复现路径。

后续如果继续完善，我会优先做三件事：

- 补充更细的 milestone evidence table。
- 对 README 中每张核心图补充更规范的图题和图注。
- 将 `run_pipeline.py` 的每一步输出进一步记录到日志中，增强复现可审计性。

## 4. 个人总结

这个项目让我从“完成一次课程作业”转向理解“如何做一个可复查的研究项目”。作为组长，我的主要贡献集中在项目工程化、文档整合、证据链梳理和最终提交质量控制上。通过这个过程，我学到的不只是 Python、NetworkX 或 README 写法，更重要的是如何把数据、图表、文字和结论组织成一个经得起检查的学术叙事。

如果用一句话总结我的收获：文献计量学项目的核心不是把图做出来，而是让每一张图都能回答一个清楚的问题，并且让每一个结论都能回到数据、方法和证据。
